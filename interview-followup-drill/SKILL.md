# Interview Follow-up Drill

把“我知道可能被问什么”推进到“我能在面试官连续质疑下，把同一个项目讲完整”。本 skill 不重复做全套备面，而是专门训练项目深挖、数据追问、方案取舍和失败复盘。

## 输入与入口

按信息完整度选择入口；优先复用已有材料，不要求用户重复提供。

| 入口 | 用户提供 | 适用情况 |
|---|---|---|
| 接续备书 | `interview-prep-brief` 的飞书链接、Markdown 或摘要 | 已经有岗位能力模型和高概率题，需要继续练追问 |
| 项目深挖 | JD + 简历中的一段项目经历 | 有明确投递岗位，想把一个经历讲透 |
| 现场补救 | 面试中被问的问题、录音转写或复盘笔记 | 已经卡过一次，想补齐下一轮可能追问 |
| 通用演练 | 岗位 JD + 用户想练的能力维度 | 没有完整简历，也要先获得一套可练的题库 |

可选补充：公司、面试轮次、用户已有题库/面经链接、真实数据边界、希望练的项目。信息缺失时，构造 `示例故事，可替换`，并显式标注所有需要替换的事实和数字。

## 与面试备书的配合

`interview-prep-brief` 解决“这个岗位会考什么、我该先准备什么”；本 skill 解决“某道题开始后，面试官接下来会怎么往下挖、我怎么答”。

如果用户给了面试备书，先抽取其中的岗位能力模型、P0 题、统一项目故事和已有答案；不要再从零生成另一套互相矛盾的故事。如果只给了 JD 或项目，则独立运行。

## 工作流

### 1. 建立统一项目故事

先选 1 个主故事，必要时再补 1 个备用故事。对每个故事固定以下事实：

- 业务背景、目标用户、候选人角色和协作对象
- 原始问题、目标指标、关键数据口径和时间范围
- 方案选项、最终取舍、执行动作
- 结果、未达预期部分、复盘结论

在进入题库前做一致性校验：同一个指标的基线、变化幅度、周期、分母不能在后续回答里改变。详细检查表见 [references/followup-answer-rubric.md](references/followup-answer-rubric.md)。

### 2. 生成高密度追问题库

按岗位能力模型组织题库，不要按空泛的“产品题/业务题”堆题。每个 P0 维度至少生成 4-6 个递进问题，覆盖：

1. `事实核验`：你的具体角色是什么？数据从哪里来？
2. `判断推导`：为什么认定这是核心问题？为什么先做这个？
3. `方案取舍`：有哪些替代方案？为什么没选？
4. `数据质疑`：指标是否可靠？如何归因？是否有副作用？
5. `失败与复盘`：哪一步没做好？重来会改什么？
6. `岗位迁移`：这段经验如何落到目标公司的业务里？

题目必须标注来源：

- `真题参考`：来自用户提供的题库/面经，且可追溯
- `相似题迁移`：基于同类岗位常见问法改写
- `JD 推演题`：从 JD 职责和能力要求推导

不要把推演题包装成真题。题目多不等于有效，优先给可形成连续压力测试的题链。

### 3. 为每道 P0/P1 题同时给两种答案

每道核心题必须包含：

- `面试官在验证什么`
- `回答思路`：结论、依据、动作、结果、反思的口头组织法
- `直接可用参考答案`：60-120 秒、适合朗读的示范稿
- `可替换位置`：用户需要改成自己事实的公司、项目、数据、角色
- `继续追问`：至少 2 个面试官可能的质疑，以及对应直接回答

直接可用答案要求：

- 先给结论，再讲 2-3 个关键动作，最后回到结果或岗位匹配。
- 每段至少包含 2 个有口径的数据；没有真实数据时使用 `[待确认]` 占位，同时给 `示例口径`，绝不把示例写成用户真实成绩。
- 同一条题链只能使用同一个项目故事、同一套数字和同一角色边界。
- 语言像候选人现场回答，不写成课程讲义或万能作文。
- 面对“你真的做了吗”类问题，要说清自己做了什么、谁参与、如何决策、怎样验证。

### 4. 模拟连续面试

至少输出一段 4-6 轮的连续对话：

`面试官提问 -> 候选人回答 -> 面试官质疑/追问 -> 候选人回答`。

追问要逐步加压：先问事实，再问判断，再挑战数据和方案，最后问失败与迁移。每轮后给一句点评，最后提供“更稳的一版回答”。

### 5. 调用题库与外部机制

如果用户提供了题库/飞书链接，优先把它作为题目和拆解思路的参考；默认可尝试读取《大厂面试真题解析》：

`https://gte09oerz5.feishu.cn/wiki/E5UxwObpMiwyfVk6j0BcsMS7nGc`

题库不可访问时继续完成任务，并说明题源降级为 `相似题迁移` 与 `JD 推演题`。

当岗位或项目类型很陌生、本地题库无法覆盖关键深挖维度，或输出题链明显泛化时，再去 GitHub 查 mock interview、interview prep、question generator 项目。学习它们的机制（题目分层、追问树、评分反馈、岗位差异），不照抄题目；高星只作为热度信号，不是采用结论。

### 6. 输出与交付

完整交付按以下结构生成：

1. `训练说明`：本次输入、题源构成、脱敏与示例边界
2. `岗位与项目速读`：本次到底在训练什么
3. `统一项目故事卡`：背景、角色、问题、策略、数据、复盘
4. `追问题库地图`：能力维度、题目、题源、优先级、故事锚点
5. `P0 追问拆解与直接可用答案`
6. `连续模拟面试`
7. `评分表与薄弱点`
8. `下一轮练习计划`

优先创建飞书文档：使用 `lark-cli docs +create --as user` 写入 XML，表格用于题目地图、评分表和素材缺口。用户没有飞书 CLI、没有权限或创建失败时，自动在当前工作目录创建 `interview-followup-drills/<岗位>-面试追问训练.md`，不要因飞书卡住。

飞书标题建议：`<公司><岗位> 面试追问训练`。

## 交付前自检

- 是否有 1 个真正统一的项目故事，而非每题一个新案例？
- 是否给了丰富题库，但 P0 题仍明确、可优先练？
- 每道 P0/P1 题是否同时有回答思路和可直接开口版？
- 数据是否有基线、结果、时间范围和口径，且前后不矛盾？
- 是否区分真实材料、示例口径和 JD 推演？
- 连续模拟是否真的逐步加压，而不是把独立问答拼在一起？
- 用户是否能今天就拿着文档朗读、录音、复盘？
